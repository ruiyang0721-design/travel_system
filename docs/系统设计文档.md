# 智能旅游推荐系统 — 系统设计文档

## 1. 需求分析

### 1.1 项目背景

随着国民经济的持续增长和人民生活水平的不断提高，旅游业已成为我国经济发展的重要支柱产业。据文化和旅游部数据显示，2024 年国内旅游人次超过 56 亿，旅游消费市场规模持续扩大。然而，伴随着旅游市场的蓬勃发展，游客在规划行程时面临着日益严重的信息过载问题。

传统的旅游行程规划方式主要依赖人工查阅攻略、对比信息、手动安排路线，这一过程通常耗时耗力，且难以综合考虑地理位置、时间约束、个人偏好、预算限制等多维因素。市面上虽已存在部分旅游推荐产品，但大多存在推荐同质化严重、缺乏个性化定制、路线规划不够智能等问题。

本项目旨在开发一个智能旅游推荐系统，通过融合 K-means 空间聚类算法和暴力穷举路径优化，实现从景点推荐到路线优化的全自动化行程规划，为用户提供高效、个性化的旅行体验。

### 1.2 项目目标

本系统的核心目标包括：

1. **智能化**：用户只需输入最少的信息（城市、天数、兴趣偏好），系统即可自动生成完整的行程方案
2. **最优化**：通过 K-means 聚类保证每日景点的地理聚集性，通过暴力穷举优化游览顺序，最小化行程总距离
3. **可视化**：在地图上直观展示行程轨迹，支持路线回放和标记点展示
4. **交互性**：支持拖拽调整行程顺序、移除景点、实时地图联动
5. **综合性**：整合预算计算、攻略分享、旅行日记等辅助功能

### 2.4 通信机制

前后端通过 RESTful API 进行通信，数据格式为 JSON。

**请求流程：**

```
1. 用户操作触发前端事件
2. Axios 发送 HTTP 请求到后端 API
3. Django URL 路由分发到对应视图函数
4. 视图函数处理业务逻辑（可能调用算法引擎）
5. 通过 Serializer 序列化数据
6. 返回 JSON 格式响应
7. 前端接收响应并更新页面
```

**认证流程：**

```
1. 用户提交用户名/密码到 /api/login/
2. 后端验证凭据，返回 JWT Token（access + refresh）
3. 前端将 Token 存储到 localStorage
4. 后续请求在 Header 中携带 Authorization: Bearer <token>
5. SimpleJWT 中间件自动验证 Token 有效性
6. 过期后使用 refresh Token 刷新
```

## 4. 接口设计

### 4.1 接口规范

系统 API 遵循 RESTful 设计规范：

- 使用标准 HTTP 方法（GET / POST / PUT / DELETE）
- JSON 格式数据交换
- JWT Token 无状态认证
- 统一响应格式：

```json
// 成功响应
{
    "status": "success",
    "data": { ... }
}

// 错误响应
{
    "status": "error",
    "message": "错误描述"
}
```

### 4.2 接口列表

#### 4.2.1 景点相关接口

| 方法 | 路径 | 认证 | 说明 | 参数 |
|------|------|------|------|------|
| GET | `/api/spots/` | 否 | 获取景点列表 | search, tag, city（查询参数） |
| GET | `/api/spots/<id>/` | 否 | 获取景点详情 | id（路径参数） |
| GET | `/api/spots/<id>/tips/` | 否 | 获取景点攻略 | id（路径参数） |
| GET | `/api/cities/` | 否 | 获取城市列表 | 无 |
| GET | `/api/city-stats/` | 否 | 获取城市统计 | city（查询参数） |

## 5. 算法设计

### 5.1 候选景点召回算法

**目标：** 从景点库中筛选出符合用户偏好的候选景点

**算法流程：**

```
输入：用户标签集合 user_tags, 游玩天数 travel_days, 目标城市 target_city, 当前用户 user
输出：候选景点列表 candidates

1. 从数据库查询属于 target_city 的所有景点
2. 初始化空列表 scored_spots
3. 对于每个景点 spot:
    a. 如果 spot.duration > 8.0，跳过（需整天游玩的主题乐园）
    b. 计算标签匹配率：match_score = |user_tags ∩ spot_tags| / |user_tags|
    c. 计算基础评分：base_score = 0.7 × match_score + 0.3 × (spot.rating / 5.0)
    d. 将 (spot, base_score) 加入 scored_spots
4. 调用协同过滤模块，计算每个候选景点的 CF 分数
5. 融合评分：
    - 有 CF 数据时：final_score = 0.6 × base_score + 0.4 × cf_score
    - 无 CF 数据时（匿名用户或数据不足）：final_score = base_score
6. 按 final_score 降序排序
7. 基于时间预算动态截断：累加景点游玩时长，直到总时长超过 travel_days × 8 小时
8. 返回候选景点列表
```

**评分公式详解：**

```
base_score = 0.7 × tag_match_ratio + 0.3 × rating_normalized
final_score = 0.6 × base_score + 0.4 × cf_score（有 CF 数据时）
```

- `tag_match_ratio`：用户选择的标签与景点标签的交集大小 / 用户选择的标签数量
  - 作用：确保推荐的景点符合用户兴趣偏好
  - 权重 0.7：兴趣匹配是最重要的筛选标准
- `rating_normalized`：景点评分 / 5.0
  - 作用：在同等标签匹配度下，优先推荐高评分景点
  - 权重 0.3：评分作为辅助参考
- `cf_score`：协同过滤分数（详见 5.2 节）
  - 作用：利用其他用户的收藏行为，为当前用户发现可能感兴趣但标签未覆盖的景点
  - 权重 0.4：与基础评分互补，提升推荐的个性化程度

**时间预算截断策略：**

与固定截取 `travel_days × 3` 个景点不同，系统采用基于时间预算的动态截断：

```
总时间预算 = 游玩天数 × 8 小时/天
依次累加候选景点的游玩时长，直到累计时长超过预算
```

这种策略的优势：
- 1 天行程自动选择较少的景点（约 8 小时游玩量）
- 5 天行程自动选择较多的景点（约 40 小时游玩量）
- 避免了固定数量可能导致的"景点过多排不下"或"景点过少不够玩"的问题

### 5.2 基于物品的协同过滤算法（Item-based CF）

**目标：** 利用其他用户的收藏行为，为当前用户发现可能感兴趣的候选景点

**算法原理：**

协同过滤模块通过分析用户-景点的收藏交互数据，计算景点之间的相似度，为当前用户预测未访问景点的兴趣分数。

**算法流程：**

```
输入：当前用户 user, 目标城市 target_city, 候选景点 ID 列表 candidate_spot_ids
输出：CF 分数字典 {spot_id: cf_score}，分数范围 [0, 1]

前置条件检查：
  - 用户已登录（非匿名）
  - 用户有该城市的收藏记录
  - 该城市至少有 2 个不同用户的收藏数据
  任一不满足则返回空字典，退化为基础评分方案

1. 构建用户-景点交互矩阵
   - 获取该城市所有收藏记录，构建 user → set(spot_id) 映射
   - 构建二元矩阵（行为用户，列为景点，值为 1 表示收藏过）

2. 计算景点间余弦相似度
   - 将矩阵转置为物品矩阵（每行是一个景点的用户向量）
   - L2 归一化后计算余弦相似度：sim(a,b) = (a·b) / (|a| × |b|)
   - 结果为 n_spots × n_spots 的相似度方阵

3. 为当前用户预测 CF 分数
   - 对每个候选景点，计算它与用户已收藏景点的平均余弦相似度
   - 跳过已收藏的景点（不需要再推荐）

4. 归一化到 [0, 1] 范围
   - 将最大 CF 分数映射为 1.0，其余按比例缩放
```

**融合策略：**

当 CF 数据可用时，候选景点的最终评分为：

```
final_score = 0.6 × base_score + 0.4 × cf_score
```

- `base_score`（权重 0.6）：基于标签匹配和评分的基础分，反映景点本身的质量和与用户偏好的匹配度
- `cf_score`（权重 0.4）：协同过滤分数，反映"与当前用户品味相似的其他用户"的收藏行为

当 CF 数据不可用时（匿名用户、数据不足），系统自动退化为基础评分方案，确保功能不受影响。

### 5.3 K-means 空间聚类 + 时间容量均衡算法

**目标：** 将候选景点按地理位置分成 K 组（K = 游玩天数），每组对应一天的行程，并确保每天的游玩时长不超过时间预算

#### 5.3.1 K-means 空间聚类

**算法流程：**

```
输入：候选景点列表 candidates, 游玩天数 travel_days
输出：按天分组的景点字典 daily_clusters

1. 边界处理：k_clusters = min(len(candidates), travel_days)
2. 提取坐标矩阵：coords = [[spot.longitude, spot.latitude] for spot in candidates]
3. 初始化 KMeans 模型：KMeans(n_clusters=k_clusters, n_init=10, random_state=42)
4. 执行聚类：labels = kmeans.fit_predict(coords)
5. 按标签分组：
    daily_clusters = {}
    对于每个景点 candidates[i]:
        cluster_index = labels[i]
        将景点加入 daily_clusters["第{cluster_index+1}天"]
6. 返回 daily_clusters
```

**参数设置说明：**

| 参数 | 值 | 说明 |
|------|------|------|
| n_clusters | travel_days | 聚类数等于游玩天数 |
| n_init | 10 | 运行 10 次取最优结果，避免局部最优 |
| random_state | 42 | 固定随机种子，保证结果可复现 |

**算法优势：**
- K-means 算法时间复杂度 O(n × K × t)，其中 n 为景点数，K 为聚类数，t 为迭代次数
- 对于本系统的数据规模（82 个景点），计算时间在毫秒级别
- 聚类结果保证每天的景点在地理上聚集，减少不必要的往返

#### 5.3.2 时间容量均衡

K-means 仅考虑地理位置，不考虑游玩时长，可能导致某一天集中了多个大景点（总时长超过 8 小时）。为此，系统在聚类后增加了时间容量均衡阶段。

**均衡流程：**

```
输入：K-means 聚类结果 daily_clusters
输出：均衡后的 daily_clusters

1. 最多迭代 10 轮
2. 每轮：
   a. 计算每个簇的质心（经纬度平均值）和总游玩时长
   b. 找出超载的簇（总时长 > 8 小时）
   c. 如果没有超载簇，均衡完成，退出
   d. 对每个超载簇：
      - 按景点到质心的距离从远到近排序
      - 逐个取出离质心最远的景点（对当天路线紧凑性影响最小）
      - 找最近的、还有空余容量的簇，将景点转移过去
      - 如果所有簇都满了，放回原簇避免景点丢失
3. 返回均衡后的分组
```

**转移策略说明：**
- 优先转出离质心最远的景点：该景点对当天路线紧凑性影响最小
- 目标簇选择：找距离最近的、尚未超载的簇，减少地理跨度
- 允许轻微超出：目标簇的总时长可以略超 8 小时，避免过于严格导致频繁转移

### 5.4 暴力穷举路径优化

**目标：** 为每天的景点组找到最短游览路线

**算法选择理由：**

每天的景点数不超过 5 个（受 8 小时时间预算约束），5! = 120 种全排列，暴力穷举可在毫秒级内求出精确最优解，比遗传算法更快、更准、代码更简洁。

#### 5.4.1 算法流程

```
输入：每天的景点列表 day_spots, 酒店节点 hotel_node（可选）
输出：最优路线

1. 如果景点数 ≤ 2，直接返回（无需穷举）
2. 枚举所有景点排列（itertools.permutations）
3. 对每种排列计算总距离：
   - 有酒店时：full_route = [酒店] + 排列 + [酒店]
   - 无酒店时：full_route = 排列
   - total_distance = Σ distance(route[i], route[i+1])
4. 取总距离最短的排列作为最优路线
```

#### 5.4.2 距离计算

```python
distance(a, b) = sqrt((lng_a - lng_b)² + (lat_a - lat_b)²)  # 欧氏距离
```

兼容处理：
- Spot 模型对象：通过 `.latitude`/`.longitude` 属性取值
- 字典对象（酒店节点）：通过 `['latitude']`/`['longitude']` 键取值

#### 5.4.3 酒店锚点约束

当用户指定了酒店时，系统将酒店作为固定锚点：

```
无酒店：直接优化景点排列 [A, B, C, D]
有酒店：酒店固定首尾 [酒店, 优化后的景点排列, 酒店]

具体实现：
- 枚举景点的全排列
- 对每种排列，在首尾添加酒店节点
- 计算包含酒店的闭环总距离
- 取总距离最短的方案
```

**未指定酒店时的处理：**
```
计算所有候选景点的经纬度平均值作为质心
avg_lat = Σ spot.latitude / len(candidates)
avg_lng = Σ spot.longitude / len(candidates)
将质心作为推荐住宿区域
```
