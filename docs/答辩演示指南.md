# 答辩演示指南

## 演示流程建议（12-15 分钟）

---

### 一、开场（1 分钟）

> "各位老师好，我的毕业设计题目是《基于 Vue 3 和 Django 的智能旅游推荐系统设计与实现》。系统采用前后端分离架构，前端使用 Vue 3 + Element Plus + 高德地图，后端使用 Django + Django REST Framework + MySQL。核心创新点是将 K-means 空间聚类算法、暴力穷举路径优化和基于物品的协同过滤算法相结合，实现从景点推荐到路线优化的全自动化行程规划。"

---

### 二、项目背景与需求（2 分钟）

**行业背景：**
- 2024 年国内旅游人次超过 56 亿，旅游市场规模持续扩大
- 游客规划行程面临信息过载，传统方式耗时耗力
- 市面产品推荐同质化严重，缺乏个性化定制

**系统目标：**
- 输入最少信息（城市、天数、兴趣）→ 自动生成完整行程
- K-means 聚类保证每日景点地理聚集，暴力穷举优化游览顺序
- 支持拖拽调整、实时地图联动、预算计算等辅助功能
- 接入 DeepSeek 大模型，支持对话式需求收集

**目标用户：**
- 自由行游客：需要智能规划行程
- 旅游爱好者：收藏景点、分享攻略、记录旅行日记
- 系统管理员：通过管理后台维护景点数据

### 三、技术架构（2 分钟）

展示架构图：

```

│          前端 Vue 3 + Element Plus           │
│          高德地图 JS API 2.0                 │
├─────────────────────────────────────────────┤
│          RESTful API (Django DRF)            │
│          JWT 认证 · CORS 跨域                │
├─────────────────────────────────────────────┤
│          算法引擎 (Python)                   │
│  K-means 聚类 · 暴力穷举 · 协同过滤(CF)    │

│          数据库 MySQL                        │
│          82 个景点 · 7 个城市                │
└─────────────────────────────────────────────┘
```

**技术选型理由：**
- **Vue 3 Composition API**：逻辑复用性强，代码组织清晰，响应式系统性能优异
- **Django REST Framework**：快速构建标准化 API，内置序列化、认证、权限管理
- **高德地图 JS API 2.0**：国内最成熟的地图服务，POI 数据丰富，路线绘制 API 完善
- **scikit-learn (KMeans)**：工业级机器学习库，算法实现成熟可靠
- **SimpleJWT**：无状态认证，适合前后端分离架构
- **SimpleUI**：Django Admin 美化主题，内置数据看板功能
- **MySQL**：关系型数据库，支持 utf8mb4 字符集，性能优异

**算法讲解**（重点，配合流程图）：

> "接下来我介绍一下核心的路线规划算法。整个流程分为三步：
>
> **第一步，候选景点召回**。系统根据用户选择的城市和兴趣标签，从数据库中筛选景点。评分公式是 `base_score = 0.7 × 标签匹配率 + 0.3 × 评分归一化`。同时，系统还会调用协同过滤模块，利用其他用户的收藏数据计算景点间的余弦相似度，为当前用户预测 CF 分数。最终评分为 `final_score = 0.6 × base_score + 0.4 × cf_score`，在标签匹配的基础上融入了用户行为分析。截断策略采用时间预算方式，累加景点游玩时长直到超过天数×8小时，自动适配不同天数的行程。
>
> **第二步，空间聚类**。使用 K-means 算法，以景点的经纬度为特征，将候选景点分成 K 组，K 等于游玩天数。这样每天的景点在地理上是聚集的，避免用户来回奔波。聚类完成后，系统还会执行时间容量均衡：检查每天的总游玩时长，如果超过 8 小时就将多余的景点转移到最近的空闲天，确保每天行程不会过于紧凑。
>
> **第三步，路径优化**。对每一天的景点组，使用暴力穷举求解最短游览路径。由于每天的景点数受时间预算约束不超过 5 个，5 的全排列只有 120 种，暴力穷举可以在毫秒级内求出精确最优解，比遗传算法更快更准。算法枚举所有排列，计算每种排列的总距离，取最短路线。
>
> 如果用户指定了酒店，系统会将酒店作为固定锚点，生成 '酒店→景点→酒店' 的闭环路线。如果没有指定酒店，系统会计算所有候选景点的质心，推荐一个住宿区域。"

### 六、答辩常见问题准备

#### Q1：为什么选择 K-means 而不是其他聚类算法？

> K-means 有三个优势适合本场景：第一，实现简单、计算效率高，对于 82 个景点的数据规模，聚类在毫秒级完成；第二，本场景中聚类数 K（天数）是用户明确指定的，K-means 需要预设 K 的特点恰好符合需求；第三，K-means 的聚类结果具有良好的可解释性，每个簇代表一天的行程，便于用户理解。DBSCAN 等算法不需要预设 K，但聚类数量不可控，不适合按天分组的需求。聚类后我们还增加了时间容量均衡阶段，解决了 K-means 只考虑空间不考虑时间的问题。

#### Q2：为什么选择暴力穷举而不是遗传算法？

> 每天的景点数受时间预算约束不超过 5 个，5 的全排列只有 120 种。暴力穷举可以在毫秒级内枚举所有排列并求出精确最优解，而遗传算法是近似解，还需要调参（种群大小、迭代代数、变异率等）。对于本场景的数据规模，暴力穷举比遗传算法更快、更准、代码更简洁。如果每天的景点数超过 10 个，遗传算法会更有优势，但在当前约束下暴力穷举是更优的选择。

#### Q3：如何处理景点数量不足的情况？

> 在三个层面做了边界保护：第一，在 candidates.py 中，如果某城市没有景点数据，直接返回空列表；候选景点采用时间预算截断（天数×8小时），自动适配不同天数。第二，在 clustering.py 中，使用 `min(len(candidates), travel_days)` 确保 K 不超过景点数，如果景点数少于天数，部分天会只有 1 个景点；聚类后还有时间容量均衡阶段，超载的景点会转移到空闲天。第三，在 routing.py 中，如果景点数 ≤ 2，直接返回不使用穷举，避免因样本太少而不必要计算。

#### Q4：前端和后端如何通信？

> 前端通过 Axios 发送 HTTP 请求到后端的 RESTful API。对于需要认证的接口，用户登录后将 JWT Token 存储在 localStorage，后续请求在 Header 中携带 `Authorization: Bearer <token>`。后端 SimpleJWT 中间件自动验证 Token 有效性。AI 对话接口使用 SSE（Server-Sent Events）实现流式输出，前端通过 EventSource 接收实时数据。

#### Q5：地图如何实现动态更新？

> 使用 Vue 3 的响应式机制。MapContainer 组件通过 `watch` 监听 itinerary 数据变化，当用户拖拽排序或移除景点时，数据变化触发 watch 回调，自动调用 `drawRoutes()` 方法重新绘制路线。同时，父组件通过递增 `:key` 属性强制 MapContainer 重新渲染，实现双重保障。

#### Q6：系统的安全性如何保证？

> 安全措施包括：密码使用 Django 的 `make_password()` 加密存储（PBKDF2 算法，SHA256 哈希）；API 使用 JWT 无状态认证，Access Token 有效期 1 天；Django ORM 自动防 SQL 注入；DRF Serializer 校验所有输入参数；CORS 中间件控制跨域访问。

#### Q7：数据库为什么选择 MySQL？

> MySQL 是最流行的开源关系型数据库，支持 utf8mb4 字符集，能很好地处理中文和特殊字符。相比 SQLite，MySQL 提供了更好的并发性能和数据完整性保障。Django 的 ORM 层提供了数据库抽象，模型定义和查询代码与具体数据库无关。本系统数据量在百级别（82 个景点），MySQL 的性能完全够用，同时为后续扩展到更大规模的数据集打下了基础。

#### Q8：K-means 聚类和路径优化的时间复杂度是多少？

> K-means 的时间复杂度是 O(n × K × t)，其中 n 是景点数，K 是聚类数（天数），t 是迭代次数。对于本系统（n=24, K=3, t≈10），计算时间在毫秒级。路径优化使用暴力穷举，时间复杂度是 O(n!)，其中 n 是每天的景点数。由于每天景点数不超过 5 个（受 8 小时时间预算约束），5! = 120 种排列，计算时间在毫秒级。总计算时间远低于 3 秒的性能要求。

#### Q9：协同过滤算法是如何工作的？

> 系统采用基于物品的协同过滤（Item-based CF）。首先从收藏表构建用户-景点交互矩阵，然后对矩阵转置计算景点间的余弦相似度。对当前用户未收藏的候选景点，计算它与用户已收藏景点的平均相似度作为 CF 分数，归一化到 [0, 1] 范围。最终候选评分公式为 `0.6 × 基础分 + 0.4 × CF 分`。当用户未登录或收藏数据不足时，系统自动退化为基础评分方案，确保功能不受影响。这种设计让系统既能利用标签匹配的确定性，又能利用用户行为数据发现潜在兴趣。

#### Q10：AI 对话功能是如何实现的？

> 后端接入 DeepSeek 大语言模型 API，通过 System Prompt 定义 AI 为"旅行需求收集助手"，明确其职责是收集参数而非生成行程。当用户消息中包含城市关键词时，系统自动从数据库提取该城市的景点数据作为上下文注入。AI 通过多轮对话收集城市、天数、偏好标签等参数，参数完整后输出结构化 JSON，前端检测到 JSON 后自动调用规划引擎。

#### Q11：管理后台的数据看板是如何实现的？

> 基于 Django SimpleUI 框架，在 admin.py 中定义 dashboard_view 视图函数，使用 Django ORM 聚合查询统计数据（景点数量、用户数量、日记数量、城市分布、标签分布等），渲染自定义的 dashboard.html 模板。通过 SIMPLEUI_CONFIG 配置将数据看板设置为左侧菜单的顶级项，与景点列表、攻略管理等并列显示。
