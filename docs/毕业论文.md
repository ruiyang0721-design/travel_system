# 基于Vue 3和Django的智能旅游推荐系统设计与实现

---

## 摘  要

随着国内旅游业的快速发展，游客在规划行程时面临信息过载的困扰。传统的人工规划方式效率低下，且难以综合考虑地理位置、兴趣偏好、时间约束等多维因素。为解决这一问题，本文设计并实现了一个基于Vue 3和Django的智能旅游推荐系统。

系统采用前后端分离的B/S架构，前端基于Vue 3框架，结合Element Plus组件库和高德地图JS API 2.0实现用户界面和地图可视化；后端基于Django 5.2框架和Django REST Framework构建RESTful API，使用SimpleJWT实现无状态认证。系统的核心创新在于路线规划算法的设计：首先通过标签匹配评分和评分归一化进行候选景点召回，并融合基于物品的协同过滤（Item-based CF）算法提升推荐个性化程度；然后利用K-means空间聚类算法将候选景点按地理位置分组，并通过时间容量均衡策略确保每日行程合理；最后通过暴力穷举对每天的游览顺序进行路径优化。此外，系统还支持酒店锚点约束、拖拽式行程编辑、AI对话式规划、预算计算、旅行日记、景点攻略等功能。

系统内置82个景点数据，覆盖北京、上海、成都、西安、杭州、重庆、广州7个城市。测试结果表明，系统能够在3秒内完成路线规划，聚类结果具有良好的地理合理性，暴力穷举路径优化能够在毫秒级内求出精确最优解。本系统为旅游行程规划提供了一种智能化、自动化的解决方案。

**关键词：** 智能旅游推荐；Vue 3；Django；K-means聚类；暴力穷举；前后端分离

## Abstract

With the rapid development of China's tourism industry, tourists face the challenge of information overload when planning their trips. Traditional manual planning methods are inefficient and struggle to comprehensively consider multiple factors such as geographic location, personal preferences, and time constraints. To address this problem, this paper designs and implements an intelligent travel recommendation system based on Vue 3 and Django.

The system adopts a front-end and back-end separated B/S architecture. The front-end is built on the Vue 3 framework, combining the Element Plus component library and AMap JS API 2.0 for user interface and map visualization. The back-end is based on the Django 5.2 framework and Django REST Framework to build RESTful APIs, using SimpleJWT for stateless authentication. The core innovation of the system lies in the route planning algorithm design: first, candidate spots are retrieved through tag matching scoring and rating normalization, fused with Item-based Collaborative Filtering (CF) to enhance personalization; then, the K-means spatial clustering algorithm groups candidate spots by geographic location with time-capacity balancing; finally, the genetic algorithm optimizes the daily tour order. Additionally, the system supports hotel anchor constraints, drag-and-drop itinerary editing, AI conversational planning, budget calculation, weather queries, travel diaries, and spot guides.

The system includes 82 spot data covering 7 cities: Beijing, Shanghai, Chengdu, Xi'an, Hangzhou, Chongqing, and Guangzhou. Test results show that the system can complete route planning within 3 seconds, the clustering results have good geographic rationality, and the genetic algorithm path optimization can effectively reduce travel distance. This system provides an intelligent and automated solution for travel itinerary planning.

**Keywords:** Intelligent Travel Recommendation; Vue 3; Django; K-means Clustering; Genetic Algorithm; Front-end and Back-end Separation

## 第1章 绪论

### 1.1 研究背景与意义

随着国民经济的持续增长和人民生活水平的不断提高，旅游业已成为我国经济发展的重要支柱产业。据文化和旅游部统计数据显示，2024年国内旅游人次超过56亿，国内旅游收入超过5万亿元，旅游市场规模持续扩大。然而，伴随着旅游市场的蓬勃发展，游客在规划行程时面临着日益严重的信息过载问题。

传统的旅游行程规划方式主要依赖人工查阅攻略、对比信息、手动安排路线，这一过程通常耗时耗力。游客需要综合考虑景点的地理位置、开放时间、门票价格、个人兴趣偏好、游玩天数、预算限制等多维因素，这对普通游客来说是一项复杂的决策任务。市面上虽已存在部分旅游推荐产品，如携程、去哪儿、马蜂窝等，但大多存在以下问题：推荐同质化严重，不同用户看到的推荐内容相似；路线规划不够智能，未考虑景点间的地理聚集性和最优游览顺序；用户交互体验不佳，缺乏灵活的行程编辑能力。

本项目旨在开发一个智能旅游推荐系统，通过融合K-means空间聚类算法和暴力穷举路径优化，实现从景点推荐到路线优化的全自动化行程规划。用户只需输入目标城市、游玩天数和兴趣偏好，系统即可自动推荐景点、规划最优路线，并在地图上可视化展示行程轨迹。本系统的研究具有以下意义：

1. **理论意义**：将机器学习领域的K-means空间聚类算法与组合优化中的暴力穷举方法相结合，探索了组合算法在旅游推荐场景中的应用，为相关研究提供了实践参考。
2. **实践意义**：为游客提供了一种智能化、自动化的行程规划工具，降低了旅游规划的时间成本和认知负担，提升了旅游体验。
3. **技术意义**：采用前后端分离架构，实现了Vue 3 + Django的技术组合，为类似Web应用的开发提供了技术方案参考。

### 1.2 国内外研究现状

#### 1.2.1 旅游推荐系统研究现状

旅游推荐系统是推荐系统在旅游领域的应用，旨在根据用户的偏好和需求，为其推荐合适的旅游景点、路线和服务。国内外学者在旅游推荐系统方面开展了大量研究。

在协同过滤推荐方面，基于用户的历史行为和偏好进行推荐，但面临冷启动和数据稀疏问题。在基于内容的推荐方面，利用景点的属性特征（如标签、评分、价格）进行推荐，但推荐结果多样性不足。在知识图谱推荐方面，构建旅游领域知识图谱，通过语义推理进行推荐，但知识图谱构建成本高。在深度学习推荐方面，利用神经网络学习用户和景点的深层特征表示，但模型可解释性差。

本系统采用基于标签匹配和评分归一化的混合推荐策略，结合K-means聚类和暴力穷举进行路线优化，在保证推荐质量的同时具有较好的可解释性。

#### 1.2.2 路径优化算法研究现状

旅行商问题（TSP）是经典的组合优化问题，目标是找到访问所有城市的最短路径。对于小规模TSP问题，精确算法（如动态规划、分支定界）可以求得最优解，但时间复杂度为指数级。对于大规模问题，启发式算法和元启发式算法是常用的近似求解方法。

遗传算法（Genetic Algorithm, GA）是受自然选择和遗传学原理启发的元启发式算法，通过模拟生物进化过程中的选择、交叉、变异等操作，在解空间中搜索近似最优解。遗传算法具有全局搜索能力强、不依赖梯度信息、易于并行化等优点，被广泛应用于TSP问题的求解。然而，对于小规模问题（如每天不超过5个景点），暴力穷举可以在毫秒级内枚举所有排列并求出精确最优解，比遗传算法更快更准。

空间聚类算法在旅游推荐中的应用也受到关注。K-means算法通过将景点按地理位置分组，使每天的景点在地理上聚集，减少不必要的往返路程。将K-means聚类与暴力穷举结合，先聚类再优化，是一种有效的旅游路线规划策略。

#### 1.2.3 前后端分离架构研究现状

前后端分离是当前Web开发的主流架构模式。前端负责用户界面和交互逻辑，后端负责业务逻辑和数据处理，两者通过RESTful API进行通信。这种架构具有开发效率高、前后端可独立部署、技术栈灵活等优点。

Vue.js是国内最流行的前端框架之一，Vue 3引入的Composition API提供了更灵活的逻辑复用方式。Django是Python生态中最成熟的Web框架，Django REST Framework（DRF）是构建RESTful API的事实标准。Vue 3 + Django的组合在Web应用开发中具有广泛的应用基础。

## 第2章 相关技术介绍

### 2.1 Vue 3 前端框架

Vue.js是一个用于构建用户界面的渐进式JavaScript框架，由尤雨溪于2014年创建。Vue 3于2020年正式发布，引入了多项重要改进。

**Composition API**是Vue 3最重要的新特性，它允许开发者使用函数的方式组织组件逻辑，替代了Vue 2中基于选项的Options API。Composition API通过`setup()`函数或`<script setup>`语法糖，将相关的逻辑代码组织在一起，提高了代码的可读性和可复用性。

**响应式系统**方面，Vue 3使用Proxy替代了Vue 2中的Object.defineProperty，解决了Vue 2中无法检测对象属性添加和删除的限制，同时提升了性能。

**Vite**是Vue团队推出的新一代前端构建工具，基于ES Module的原生模块加载能力，实现了极速的开发服务器启动和热模块替换（HMR），相比Webpack具有显著的开发体验优势。

本系统使用Vue 3.5 + Vite 8.0进行前端开发，采用Composition API和`<script setup>`语法糖组织组件逻辑。

### 2.2 Django 后端框架

Django是一个高级Python Web框架，遵循"电池全包"（batteries included）的设计理念，内置了ORM、模板引擎、表单处理、认证系统、管理后台等丰富的功能组件。

Django的核心组件包括：

- **ORM（Object-Relational Mapping）**：将数据库表映射为Python类，通过类和对象的方式操作数据库，避免直接编写SQL语句，同时提供了数据库无关性。
- **Admin管理后台**：自动生成的管理界面，可以对数据进行增删改查操作，适合后台数据管理。
- **URL路由**：将URL模式映射到视图函数，支持路径参数和正则表达式。
- **中间件**：请求/响应处理的钩子框架，可以实现认证、日志、CORS等通用功能。
- **模板引擎**：将Python数据渲染为HTML页面。

本系统使用Django 5.2作为后端框架，利用其ORM进行数据库操作，Admin管理后台进行数据管理。

### 2.3 Django REST Framework

Django REST Framework（DRF）是构建在Django之上的Web API框架，是Django生态中构建RESTful API的事实标准。DRF提供了以下核心功能：

- **序列化器（Serializer）**：将复杂的查询集（QuerySet）和模型实例转换为Python原生数据类型，便于渲染为JSON。同时提供反序列化和数据验证功能。
- **视图集（ViewSet）和视图（APIView）**：提供基于类和基于函数的视图组织方式。
- **认证（Authentication）**：支持Session认证、Token认证、JWT认证等多种认证方式。
- **权限（Permission）**：控制哪些用户可以访问哪些API端点。
- **路由器（Router）**：自动为视图集生成URL配置。

本系统使用DRF构建RESTful API，采用Serializer进行数据序列化和验证，使用基于函数的视图组织业务逻辑。

### 2.4 高德地图 JS API

高德地图是国内领先的数字地图内容、导航和位置服务解决方案提供商。高德地图JS API 2.0是基于HTML5的地图JavaScript API，提供了丰富的地图功能：

本系统使用高德地图JS API 2.0实现地图展示、POI搜索（酒店联想输入）、路线绘制和标记点展示等功能。

### 2.5 K-means 聚类算法

K-means是一种经典的无监督学习算法，用于将数据集划分为K个簇。算法的基本思想是：通过迭代优化，使每个数据点到其所属簇中心的距离之和最小。

**算法流程：**

```
输入：数据集 D = {x₁, x₂, ..., xₙ}，聚类数 K
输出：K 个簇 C₁, C₂, ..., Cₖ

1. 随机选择 K 个数据点作为初始聚类中心 μ₁, μ₂, ..., μₖ
2. 重复以下步骤直到收敛：
   a. 分配步骤：对于每个数据点 xᵢ，计算其到各聚类中心的距离，
      将其分配到最近的聚类中心所属的簇
      Cⱼ = {xᵢ : ||xᵢ - μⱼ|| ≤ ||xᵢ - μₗ||, ∀l ≠ j}
   b. 更新步骤：重新计算每个簇的聚类中心
      μⱼ = (1/|Cⱼ|) Σ xᵢ (xᵢ ∈ Cⱼ)
3. 返回聚类结果
```

**时间复杂度**：O(n × K × t)，其中n为数据点数量，K为聚类数，t为迭代次数。

**参数设置**：
- `n_init=10`：运行10次取最优结果，避免因初始中心选择不当而陷入局部最优。
- `random_state=42`：固定随机种子，保证结果可复现。

在本系统中，K-means算法用于将候选景点按地理位置分组，K等于游玩天数，每个簇代表一天的行程。

### 2.6 暴力穷举算法

暴力穷举（Brute Force）是一种通过枚举所有可能的解来寻找最优解的算法。对于排列问题，暴力穷举通过生成所有排列并逐一评估，保证找到全局最优解。

**适用场景：**

当问题规模较小时，暴力穷举具有以下优势：
- **精确性**：枚举所有可能，保证找到全局最优解，不存在近似误差
- **简洁性**：实现简单，不需要调参，代码可维护性好
- **高效性**：对于小规模问题，计算时间在可接受范围内

以本系统为例，每天的景点数受时间预算约束不超过5个，5! = 120种全排列，暴力穷举可在毫秒级内完成计算，比遗传算法等近似算法更快更准。

**基本流程：**

```
1. 确定待排列的元素集合
2. 使用 itertools.permutations 生成所有排列
3. 对每种排列计算目标函数值（如总距离）
4. 取目标函数值最优的排列作为最终解
```

**与遗传算法的对比：**

| 对比维度 | 暴力穷举 | 遗传算法 |
|----------|----------|----------|
| 解的质量 | 精确最优解 | 近似最优解 |
| 时间复杂度 | O(n!) | O(G × P × n) |
| 适用规模 | n ≤ 10 | n > 10 |
| 参数调优 | 无需调参 | 需调整多个参数 |
| 实现难度 | 简单 | 较复杂 |

对于本系统的场景（每天景点数 ≤ 5），暴力穷举在精度、速度和代码简洁性上均优于遗传算法。

**时间复杂度**：O(n!)，其中n为每天的景点数。当n=5时，5!=120，计算时间在微秒级。

### 2.7 JWT 认证机制

JSON Web Token（JWT）是一种开放标准（RFC 7519），用于在各方之间安全地传输信息。JWT由三部分组成，以点号分隔：

```
Header.Payload.Signature
```

- **Header**：包含令牌类型（JWT）和签名算法（如HS256）。
- **Payload**：包含声明（Claims），如用户ID、过期时间等。
- **Signature**：对Header和Payload的签名，用于验证令牌的完整性。

**JWT认证流程：**

```
1. 用户提交用户名和密码
2. 服务器验证凭据，生成JWT Token
3. 客户端存储Token（localStorage）
4. 后续请求在Header中携带：Authorization: Bearer <token>
5. 服务器验证Token的有效性
6. Token过期后使用Refresh Token刷新
```

**优点**：无状态（服务器不需要存储会话信息）、可扩展、跨域友好。

本系统使用SimpleJWT库实现JWT认证，Access Token有效期5分钟，Refresh Token有效期1天。

### 2.8 Element Plus

Element Plus是基于Vue 3的组件库，由饿了么前端团队开发维护。它提供了60+个高质量组件，包括按钮、表单、表格、对话框、导航、布局等，覆盖了Web应用开发的常见UI需求。Element Plus具有以下特点：

- 完整的TypeScript支持
- 按需引入，支持Tree Shaking
- 丰富的主题定制能力
- 完善的国际化支持

本系统使用Element Plus 2.13构建用户界面，利用其表单、卡片、对话框、导航等组件快速搭建页面。

随着国内旅游业的快速发展，游客在规划行程时面临信息过载的困扰。传统的人工规划方式效率低下，且难以综合考虑地理位置、兴趣偏好、时间约束等多维因素。本项目旨在开发一个智能旅游推荐系统，通过算法自动为用户生成最优行程方案，降低旅游规划的时间成本和认知负担。

#### 4.4.1 候选景点召回算法

**目标**：从景点库中筛选出符合用户偏好的候选景点。

**算法流程**：

```
输入：用户标签集合 user_tags, 游玩天数 travel_days, 目标城市 target_city, 当前用户 user
输出：候选景点列表 candidates

1. 从数据库查询属于 target_city 的所有景点
2. 初始化空列表 scored_spots
3. 对于每个景点 spot:
   a. 如果 spot.duration > 8.0，跳过（需整天游玩的主题乐园等不合理景点）
   b. 计算标签匹配率：
      match_score = |user_tags ∩ spot_tags| / |user_tags|
   c. 计算基础评分：
      base_score = 0.7 × match_score + 0.3 × (spot.rating / 5.0)
   d. 将 (spot, base_score) 加入 scored_spots
4. 调用协同过滤模块（cf.py），计算每个候选景点的 CF 分数
5. 融合评分：
   - 有 CF 数据时：final_score = 0.6 × base_score + 0.4 × cf_score
   - 无 CF 数据时（匿名用户或数据不足）：final_score = base_score
6. 按 final_score 降序排序
7. 基于时间预算动态截断：累加景点游玩时长，直到总时长超过 travel_days × 8 小时
8. 返回候选景点列表
```

**评分公式**：

```
base_score = 0.7 × tag_match_ratio + 0.3 × rating_normalized
final_score = 0.6 × base_score + 0.4 × cf_score（有 CF 数据时）
```

其中：
- `tag_match_ratio`：用户选择的标签与景点标签的交集大小 / 用户选择的标签数量，衡量景点与用户偏好的契合度。权重0.7，兴趣匹配是最重要的筛选标准。
- `rating_normalized`：景点评分 / 5.0，确保在同等匹配度下优先推荐高评分景点。权重0.3，评分作为辅助参考。
- `cf_score`：基于物品的协同过滤分数（详见4.4.2节），利用其他用户的收藏行为为当前用户发现可能感兴趣的景点。权重0.4，与基础评分互补提升个性化程度。

**时间预算截断策略**：

系统采用基于时间预算的动态截断，而非固定截取`travel_days × 3`个景点：

```
总时间预算 = 游玩天数 × 8 小时/天
依次累加候选景点的游玩时长，直到累计时长超过预算
```

这种策略的优势：
- 1天行程自动选择较少的景点（约8小时游玩量）
- 5天行程自动选择较多的景点（约40小时游玩量）
- 避免了固定数量可能导致的"景点过多排不下"或"景点过少不够玩"的问题

#### 4.4.2 基于物品的协同过滤算法（Item-based CF）

**目标**：利用其他用户的收藏行为，为当前用户发现可能感兴趣的候选景点，提升推荐的个性化程度。

**算法原理**：

协同过滤模块通过分析用户-景点的收藏交互数据，计算景点之间的相似度，为当前用户预测未访问景点的兴趣分数。系统采用基于物品的协同过滤（Item-based CF），而非基于用户的方法，因为物品数量（82个景点）远少于潜在用户数量，相似度计算更稳定。

**前提条件**（任一不满足则返回空字典，退化为基础评分方案）：
- 用户已登录（非匿名）
- 用户有该城市的收藏记录
- 该城市至少有2个不同用户的收藏数据

**算法流程**：

<!-- ===== 【图4-6】协同过滤算法流程图 ===== -->
<!-- 
  【图4-6】协同过滤算法流程图
  文件名建议：fig4-6_cf_algorithm_flowchart.png
  描述：展示 Item-based CF 的完整计算流程。
  绘制建议：使用 draw.io 绘制流程图。
    开始 → 前置条件检查（用户已登录？有收藏？有2+用户？）
    → [不满足]返回空字典（退化为基础评分）
    → [满足]获取该城市所有收藏记录 → 构建 user→set(spot_id) 映射
    → 构建用户-景点二元矩阵 → 转置为物品矩阵
    → L2归一化 → 计算余弦相似度矩阵（n×n）
    → 对每个候选景点：计算与已收藏景点的平均相似度 → 跳过已收藏
    → 归一化到[0,1] → 返回CF分数字典
-->
<!-- ===== 【图4-6 结束】===== -->

```
输入：当前用户 user, 目标城市 target_city, 候选景点 ID 列表 candidate_spot_ids
输出：CF 分数字典 {spot_id: cf_score}，分数范围 [0, 1]

1. 构建用户-景点交互矩阵
   - 获取该城市所有收藏记录，构建 user → set(spot_id) 映射
   - 构建二元矩阵（行为用户，列为景点，值为1表示收藏过）

2. 计算景点间余弦相似度
   - 将矩阵转置为物品矩阵（每行是一个景点的用户向量）
   - L2 归一化后计算余弦相似度：sim(a,b) = (a·b) / (|a| × |b|)
   - 结果为 n_spots × n_spots 的相似度方阵

3. 为当前用户预测 CF 分数
   - 对每个候选景点，计算它与用户已收藏景点的平均余弦相似度
   - 跳过已收藏的景点（不需要再推荐）

4. 归一化到 [0, 1] 范围
   - 将最大 CF 分数映射为1.0，其余按比例缩放
```

**融合策略**：

当CF数据可用时，候选景点的最终评分为：

```
final_score = 0.6 × base_score + 0.4 × cf_score
```

- `base_score`（权重0.6）：基于标签匹配和评分的基础分，反映景点本身的质量和与用户偏好的匹配度
- `cf_score`（权重0.4）：协同过滤分数，反映"与当前用户品味相似的其他用户"的收藏行为

当CF数据不可用时（匿名用户、数据不足），系统自动退化为基础评分方案，确保功能不受影响。

**算法流程**：

```
输入：候选景点列表 candidates, 游玩天数 travel_days
输出：按天分组的景点字典 daily_clusters

1. 边界处理：k = min(len(candidates), travel_days)
2. 提取坐标矩阵：
   coords = [[spot.longitude, spot.latitude] for spot in candidates]
3. 初始化 KMeans 模型：
   KMeans(n_clusters=k, n_init=10, random_state=42)
4. 执行聚类：labels = kmeans.fit_predict(coords)
5. 按标签分组：
   daily_clusters = {}
   for i, spot in enumerate(candidates):
       cluster_index = labels[i]
       daily_clusters["第{cluster_index+1}天"].append(spot)
6. 返回 daily_clusters
```

**参数设置**：

| 参数 | 值 | 说明 |
|------|------|------|
| n_clusters | travel_days | 聚类数等于游玩天数 |
| n_init | 10 | 运行10次取最优 |
| random_state | 42 | 固定随机种子 |

#### 4.4.4 暴力穷举路径优化

**目标**：为每天的景点组找到最短游览路线。

**算法选择理由**：每天的景点数不超过5个（受8小时时间预算约束），5! = 120种全排列，暴力穷举可在毫秒级内求出精确最优解，比遗传算法更快更准、代码更简洁。

<!-- ===== 【图4-8】暴力穷举路径优化流程图 ===== -->
<!-- 
  【图4-8】暴力穷举路径优化流程图

**距离计算**：
```
distance(a, b) = sqrt((lng_a - lng_b)² + (lat_a - lat_b)²)
```
兼容Spot模型对象（通过.latitude/.longitude属性）和字典对象（酒店节点，通过['latitude']/['longitude']键）。

**算法流程**：

```
输入：一天的景点列表 day_spots, 酒店节点 hotel_node（可选）
输出：最优路线 optimized_route

1. 边界处理：如果景点数 ≤ 2，直接返回
2. 使用 itertools.permutations 生成所有景点排列
3. 对每种排列：
   a. 有酒店时：full_route = [酒店] + 排列 + [酒店]
   b. 无酒店时：full_route = 排列
   c. 计算总距离 = Σ distance(route[i], route[i+1])
4. 返回总距离最短的排列
```

#### 4.4.5 酒店锚点约束

当用户指定了酒店时，系统将酒店作为固定锚点，生成闭环路线：

```
用户指定酒店 → 使用高德地图获取经纬度 → 构造酒店节点
未指定酒店 → 计算所有候选景点的经纬度平均值作为质心 → 推荐住宿区域
```

路径优化时，酒店固定在路线的首尾，只对中间的景点排列进行优化，确保每天的路线是"酒店→景点→酒店"的闭环。

class Spot(models.Model):

def generate_itinerary_with_hotel(user_data, hotel_location=None):
    target_city = user_data.get('city', '北京')
    travel_days = int(user_data.get('days', 3))
    user_tags = user_data.get('tags', [])

    # 1. 候选景点召回
    candidates = generate_candidates(user_tags, travel_days, target_city)
    if not candidates:
        return {}

    # 2. 空间聚类
    daily_clusters = cluster_spots(candidates, travel_days)

    # 3. 酒店节点构造
    hotel_node = None
    if hotel_location:
        hotel_node = {
            'id': 'hotel_custom',
            'name': hotel_location.get('name', '您的住宿酒店'),
            'latitude': float(hotel_location['lat']),
            'longitude': float(hotel_location['lng']),
            'is_hotel': True,
        }
    else:
        avg_lat = sum(s.latitude for s in candidates) / len(candidates)
        avg_lng = sum(s.longitude for s in candidates) / len(candidates)
        hotel_node = {
            'id': 'hotel_ai_rec',
            'name': 'AI 推荐住宿区域',
            'latitude': avg_lat,
            'longitude': avg_lng,
            'is_hotel': True,
        }

    # 4. 路径优化
    formatted_plan = {}
    for day_name, spots_in_cluster in daily_clusters.items():
        sorted_daily_route = optimize_daily_route(spots_in_cluster, hotel_node)
        formatted_plan[day_name] = sorted_daily_route

    return formatted_plan

候选景点召回模块（candidates.py）实现了标签匹配、协同过滤融合和时间预算截断：

```python
# 融合权重常量
ALPHA = 0.6  # 原始分数（标签匹配 + 评分）的权重
BETA = 0.4   # 协同过滤分数的权重
DAILY_TIME_BUDGET = 8.0  # 每日有效游玩时间预算（小时）

def generate_candidates(user_tags, travel_days, target_city, user=None):
    total_time_budget = travel_days * DAILY_TIME_BUDGET
    all_spots = Spot.objects.filter(city=target_city)
    if not all_spots.exists():
        return []

    scored_spots = []
    user_tags_set = set(user_tags) if user_tags else set()

    for spot in all_spots:
        if spot.duration > 8.0:  # 排除需整天游玩的景点
            continue
        spot_tags = set(spot.tags.replace('，', ',').split(','))
        match_score = len(spot_tags & user_tags_set) / len(user_tags_set) if user_tags_set else 0.5
        base_score = 0.7 * match_score + 0.3 * (spot.rating / 5.0)
        scored_spots.append({'spot': spot, 'base_score': base_score})

    # 协同过滤加分
    candidate_ids = [item['spot'].id for item in scored_spots]
    cf_scores = build_item_cf_scores(user, target_city, candidate_ids)

    # 融合分数
    for item in scored_spots:
        sid = item['spot'].id
        cf = cf_scores.get(sid, 0.0)
        if cf_scores:
            item['score'] = ALPHA * item['base_score'] + BETA * cf
        else:
            item['score'] = item['base_score']

    scored_spots.sort(key=lambda x: x['score'], reverse=True)

    # 基于时间预算的动态截断
    candidates = []
    accumulated_duration = 0.0
    for item in scored_spots:
        candidates.append(item['spot'])
        accumulated_duration += item['spot'].duration
        if accumulated_duration > total_time_budget:
            break
    return candidates
```

该模块的核心设计决策：
- **融合权重**：基础分占60%，CF分占40%，在保证推荐质量的同时利用协同过滤提升个性化程度。
- **时间预算截断**：相比固定截取`days×3`个景点，时间预算策略能自动适配不同天数，避免景点过多或过少。
- **降级策略**：匿名用户或CF数据不足时自动退化为基础评分方案，确保功能可用。

def cluster_spots(candidates, travel_days):
    k_clusters = min(len(candidates), travel_days)
    coords = np.array([[spot.longitude, spot.latitude] for spot in candidates])
    kmeans = KMeans(n_clusters=k_clusters, n_init=10, random_state=42)
    labels = kmeans.fit_predict(coords)

    daily_itinerary = {f"第{i+1}天": [] for i in range(k_clusters)}

        cluster_index = labels[i]
        daily_itinerary[f"第{cluster_index+1}天"].append(spot)

#### 5.4.4 暴力穷举实现

```python
import itertools
import math

def calc_distance(spot1, spot2):
    """计算两个景点之间的欧氏距离，兼容Spot对象和字典（酒店节点）"""
    lat1 = spot1['latitude'] if isinstance(spot1, dict) else spot1.latitude
    lng1 = spot1['longitude'] if isinstance(spot1, dict) else spot1.longitude
    lat2 = spot2['latitude'] if isinstance(spot2, dict) else spot2.latitude
    lng2 = spot2['longitude'] if isinstance(spot2, dict) else spot2.longitude
    return math.sqrt((lng1 - lng2)**2 + (lat1 - lat2)**2)

def _total_distance(route):
    """计算一条路线的总距离"""
    return sum(calc_distance(route[i], route[i+1]) for i in range(len(route) - 1))

def optimize_daily_route(day_spots, hotel_node=None):
    """使用暴力穷举优化一天的景点游览顺序"""
    n_spots = len(day_spots)
    if n_spots == 0:
        return []
    # 单个或两个景点，直接返回
    if n_spots <= 2:
        if hotel_node:
            return [hotel_node] + day_spots + [hotel_node]
        return day_spots

    # 暴力穷举所有排列，找最短路线
    best_route = None
    best_dist = float('inf')

    for perm in itertools.permutations(day_spots):
        if hotel_node:
            full_route = [hotel_node] + list(perm) + [hotel_node]
        else:
            full_route = list(perm)
        dist = _total_distance(full_route)
        if dist < best_dist:
            best_dist = dist
            best_route = full_route

    return best_route
```

## 第7章 总结与展望

### 7.1 工作总结

本文设计并实现了一个基于Vue 3和Django的智能旅游推荐系统。系统采用前后端分离的B/S架构，前端基于Vue 3 + Element Plus + 高德地图JS API，后端基于Django + Django REST Framework + MySQL 8.0。

本文的主要工作和成果包括：

1. **系统架构设计**：设计了前后端分离的四层架构（表现层、接口层、业务逻辑层、数据访问层），实现了模块化的系统结构，前后端可独立开发和部署。

2. **数据库设计**：设计了Spot、Favorite、TravelDiary、SpotTip、SavedPlan五个数据模型，通过外键建立了完整的实体关系，使用联合唯一约束保证数据一致性。

3. **路线规划算法设计**：设计了三阶段路线规划算法：
   - 候选景点召回：基于标签匹配评分（0.7权重）和评分归一化（0.3权重）的基础评分，融合基于物品的协同过滤（Item-based CF，0.4权重）提升个性化程度，采用时间预算动态截断策略（每天8小时预算）自动适配不同天数
   - K-means空间聚类：以经纬度为特征，K=游玩天数，配合时间容量均衡策略确保每日行程合理
   - 暴力穷举路径优化：枚举所有排列，取总距离最短的路线

4. **系统功能实现**：实现了12个功能模块、20+个API端点，覆盖了旅游行程规划的全流程，包括智能规划、景点浏览、地图可视化、拖拽编辑、收藏、日记、攻略、AI对话等。

5. **系统测试**：完成了28个功能测试用例、算法测试、性能测试和兼容性测试，系统各项指标均达到预期要求。

### 7.2 不足与展望

尽管系统已实现了预期的功能目标，但仍存在以下不足和改进空间：

1. **数据规模有限**：系统内置82个景点，覆盖7个城市，数据量较小。未来可接入携程、美团等平台的API，获取更丰富的景点数据。

2. **推荐算法可进一步优化**：当前协同过滤基于收藏数据的二元交互（收藏/未收藏），数据较为稀疏。未来可引入用户评分、浏览行为、停留时间等多维交互数据，结合深度学习模型（如NCF、Graph Neural Network）提升推荐精度。

3. **距离计算简化**：当前使用欧氏距离计算景点间的距离，未考虑实际道路情况。未来可接入高德地图的路线规划API，获取实际驾车或步行距离。

4. **缺乏移动端适配**：当前系统仅支持PC端浏览器访问。未来可进行响应式布局优化，或开发微信小程序版本。

5. **社交功能薄弱**：当前仅支持基础的收藏和日记功能。未来可增加行程分享、多人协作编辑、评论互动等社交功能。

6. **部署方案简单**：当前使用Django开发服务器和MySQL数据库。生产环境应使用Gunicorn + Nginx的部署方案，并考虑容器化（Docker）部署。

7. **AI对话能力有限**：当前AI对话仅用于收集规划参数，未实现真正的智能对话。未来可让AI直接生成行程建议，实现更自然的人机交互。
